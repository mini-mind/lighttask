# LightTask PRD

**版本**：3.0  
**日期**：2026-04-16

## 1. 产品定位

LightTask 是通用任务中心编排内核，不是完整工作流应用。

它解决的是：

- 任务定义与推进
- 任务依赖关系描述
- 调度事实计算
- 执行上下文与结果留痕

它不解决的是：

- UI
- provider / agent 适配
- 审批产品形态
- 应用专属 read model

## 2. 核心产品判断

1. `Task` 是唯一真源对象
2. `Graph` 是依赖关系视图
3. `Task` 使用设计态与执行态双轴
4. 调度直接围绕任务集合工作

## 3. 要解决的问题

旧 Graph-first 模型的问题是：

- 用户必须先把图设计完整，任务才能真正进入系统
- 设计对象与执行对象分裂成两套心智
- `publish -> materialize` 链路过长
- `draft` 与执行状态容易混淆

当前版本已经把这些问题收口为：

- 任务可以先创建，再逐步补齐设计与依赖
- 草稿任务天然阻塞，不会误入调度
- 关系图只描述约束，不再生成任务

## 4. 当前主链

```text
Plan
  -> Task[]
  -> Graph(依赖关系视图)
  -> getPlanSchedulingFacts
  -> 上层执行
  -> advanceTask
  -> Runtime / Output
```

## 5. 关键对象职责

### `Plan`

- 一轮编排的容器边界
- 承接计划级生命周期

### `Task`

- 承接任务定义、归属、状态、步骤、扩展
- 是调度、审批、执行、补偿、审计主对象

### `Graph`

- 承接依赖与约束关系
- 仅引用已归属当前计划的任务

### `Runtime`

- 承接执行上下文

### `Output`

- 承接执行结果与交付物

## 6. 用产品语言理解对象关系

如果把 LightTask 看成一个任务协作产品，可以这样理解：

- `Plan` 是一张工作台，定义这一轮协作的边界。
- `Task` 是工作台上的任务卡片，是唯一真源。
- `Graph` 是卡片之间的依赖线，只描述先后与阻塞。
- `Runtime` 是任务被执行时的运行记录。
- `Output` 是任务执行后沉淀下来的结果物。

它们之间的关系是：

1. 先创建 `Plan`
2. 再把 `Task` 放进这个 `Plan`
3. 用 `Graph` 把这些 `Task` 连成依赖关系
4. 调度器读取 `Task + Graph` 计算“谁现在能做”
5. 上层人或 Agent 推进 `Task`
6. `Runtime / Output` 记录执行过程与结果

## 7. 产品语义

### 任务归属

- 由 `task.planId` 显式声明
- Graph 只负责依赖关系视图

### 调度门槛

- `designStatus = draft`：阻塞
- `designStatus = ready`：允许进入调度

### 图发布

- 发布关系视图快照
- 推进 `plan.revision`
- 要求上层基于新依赖基线重新确认计划

### 任务同步

`materializePlanTasks` 当前只同步关系 provenance，不再把 Graph 的设计字段回写到 Task。

### 计划发射

`launchPlan` 当前只做：

- 收集已发布图上的任务快照
- 把计划从 `ready` 确认到 `confirmed`

## 8. 成功标准

1. 应用层围绕任务而不是围绕图接入
2. Graph 与 Task 不再双真源
3. 草稿任务不会误入调度
4. 文档、实现、测试对同一主链达成一致
5. 发布物可以直接对外发布
